# Phase 8：XFVerse 1.0 正式版打包、发布与文档体系总计划

Last updated: 2026-06-20

## 1. 阶段状态

- 当前阶段：Phase 8。
- 当前状态：Phase 8.4 正式打包与安装器工程已完成，下一步进入 Phase 8.5 安装说明。
- 发布目标：XFVerse 1.0.0 Windows 正式版。
- 默认约束：不 commit、不 push、不执行 database update、不新增 migration。
- 本计划是 Phase 8 的总控文档。每个小阶段开始前，必须阅读本文件及对应小阶段文档；执行完成后，必须更新对应小阶段文档、`PHASE_8_STAGE_LOG.md` 和 `PHASE_8_KNOWN_ISSUES.md`。

## 2. 阶段背景与当前证据

仓库已具备测试安装包链路，但尚不满足公开正式发布要求：

- 当前测试安装脚本为 `scripts/packaging/Build-TestInstaller.ps1`。
- 当前测试包会复制本机用户数据快照，并生成预置测试数据库。
- 当前测试安装器会删除并覆盖当前用户的应用数据目录。
- 当前测试包可能携带 API Key、Token、WebDAV 账号、播放记录和画像数据，禁止作为正式版公开发布。
- 当前应用项目没有明确配置 `Version`、`AssemblyVersion`、`FileVersion` 或 `InformationalVersion`。
- 当前设置页版本文本来自程序集版本，未形成单一版本源。
- 当前敏感字段保护实现仅是 Base64 编码，不能作为正式版凭据安全方案。
- 当前 x64 自包含测试发布目录约 1.14 GB，测试安装包约 1.07 GB，需要建立体积清单并裁剪不必要资产。
- 当前 `docs/安装说明.md` 和 `docs/使用说明.md` 内容较旧，部分入口、功能边界和页面位置已与当前代码不一致。
- 当前根 `README.md` 仅包含项目名和一句简介，不足以作为正式项目入口。
- 当前仓库仅有局部第三方声明，正式发布前必须完成随包第三方组件、许可证和来源审计。

上述问题是 Phase 8 的输入，不代表全部都必须用大范围重构解决。每项都必须先形成证据、风险判断和最小方案。

## 3. 总目标

Phase 8 负责完成以下闭环：

1. 冻结 XFVerse 1.0 的真实功能范围、系统要求、已知限制和发布边界。
2. 解决公开发布前的安全、隐私、版本、数据生命周期和启动升级阻断项。
3. 建立与内部测试包完全分离的正式版打包链路。
4. 生成不含用户数据、私有路径、账号、Token、API Key、播放记录或缓存快照的正式安装包。
5. 完成符合当前代码实际的软件设计说明书，并记录现有 UI 从草图、页面规格到实际 WPF UI 的设计和落地过程。
6. 完成安装说明、软件使用说明书、帮助文档、README、发布说明和第三方声明。
7. 在干净环境、已有数据环境和升级环境完成 RC 验收。
8. 输出可复现的 1.0.0 GA 发布产物、校验值、发布清单和回滚说明。

## 4. 正式版范围

### 4.1 正式版目标平台

- Windows 10 / Windows 11。
- 1.0 GA 只正式支持 Windows x64。
- Windows ARM64 不进入 1.0 GA；现有 ARM64 测试路径仅作为后续实验能力保留。
- 正式包采用 x64 self-contained 发布；Phase 8.4 负责验证体积、启动、原生依赖和兼容性。

### 4.2 文档语言

- 1.0 首版文档以简体中文为主。
- 安装器和应用现有主要界面为中文，正式文档应与实际中文 UI 文案一致。
- 英文 README 或英文安装器不是 1.0 必需项。

### 4.3 功能边界

文档只能描述代码和正式 RC 包中实际可用的功能，禁止根据历史计划、旧截图或未完成设计稿扩写。

至少覆盖：

- 首页。
- 媒体库。
- 影片发现。
- 电影、电视剧、季和 Episode 详情。
- 本地目录和 WebDAV 配置、扫描、识别、修正。
- 播放器、多播放源、音轨、字幕和在线字幕。
- 收藏、想看、已看、不想看、观影历史。
- 观影洞察和 Movie-only 边界。
- TMDB、OMDb、OpenSubtitles 和 OpenAI-compatible AI 配置。
- 软件缓存、视频缓存和相关清理语义。
- 主题和个人资料。
- 移出媒体库、删除记录和真实文件之间的非破坏性语义。

不得把 TV 写成已进入观影洞察、AI 推荐或 fingerprint；不得把移出媒体库写成删除真实文件。

## 5. 文档定位边界

### 5.1 README

定位：仓库和项目的入口页。

负责：

- 产品简介和核心能力。
- 支持平台与正式版本。
- 快速开始。
- 文档导航。
- 开发构建和正式打包入口。
- 数据与隐私提示。
- 第三方组件和许可证入口。
- 当前发布状态与下载入口。

不负责：

- 逐屏完整操作教程。
- 完整安装器步骤。
- 大量故障排查问答。
- 阶段开发流水账。

### 5.2 安装说明

定位：让目标用户安全完成安装、首次启动、升级、修复和卸载。

负责：

- 系统要求和架构选择。
- 下载、校验、安装步骤。
- Windows 安全提示和数字签名状态。
- 首次启动和初始化。
- 数据目录、配置目录、缓存目录的用途。
- 旧版本升级、数据迁移和备份。
- 修复安装、覆盖安装、卸载和数据保留策略。
- 安装失败和启动失败的最短排查入口。

不负责：

- 介绍全部业务功能。
- 教用户完成完整媒体库工作流。
- 收纳所有故障问题。

### 5.3 软件使用说明书

定位：按功能和业务流程完整说明软件如何使用。

负责：

- 概念、界面导航和推荐上手流程。
- 各页面、按钮、状态、筛选和批量操作。
- 本地 / WebDAV 媒体接入。
- 扫描、识别、修正和复扫。
- 媒体库、详情、播放、字幕、发现、收藏、历史、洞察。
- 设置项与第三方服务能力边界。
- 数据操作的准确语义。
- 常见完整工作流和操作结果。

不负责：

- 重复安装器的每一步。
- 按错误现象堆积排障条目。
- 描述代码中不存在的功能。

### 5.4 帮助文档

定位：用户遇到问题时，按症状、错误和任务快速查找答案。

负责：

- FAQ。
- 安装、启动、扫描、识别、播放、字幕、网络、API、缓存和性能排障。
- 错误现象、可能原因、检查步骤、恢复步骤和升级给开发者的信息。
- 日志、版本、系统信息和脱敏诊断信息的获取方式。
- 数据备份、恢复和避免误操作。

不负责：

- 从头到尾介绍软件。
- 复制整本使用说明书。
- 要求用户公开敏感配置、完整路径、完整 URL 或凭据。

### 5.5 发布说明与第三方声明

定位：说明 1.0.0 的交付范围、限制、校验值、升级注意事项和随包第三方组件。

不得把 Known Issues 隐藏在营销文案后；影响数据、安全、安装或核心播放的限制必须明确披露。

### 5.6 软件设计说明书

定位：面向开发、维护、测试和技术评审人员，说明软件当前实际采用的总体架构、模块、数据、流程、安全、部署和 UI 设计。

负责：

- 系统上下文、解决方案结构和模块边界。
- 数据模型、关键流程、外部服务和安全隐私设计。
- 性能、可靠性、诊断、部署和升级设计。
- UI 信息架构、设计系统、页面结构和交互状态。
- 记录现有 UI 的形成过程：先绘制草图和页面规格，再依据草图修改实际 UI，并通过运行验收持续收口。
- 建立现有草图、页面规格与当前 XAML / ViewModel 的追溯关系。

不负责：

- 重新绘制一套新的 UI 草图。
- 新增 UI 重做阶段。
- 代替安装说明、软件使用说明书或帮助文档。
- 把旧草图中未落地的功能写成当前实现。

## 6. 全阶段固定规则

### 6.1 发布安全

- 正式包绝不复制开发机或当前用户的应用数据。
- 正式包绝不预置私有数据库、媒体记录、播放历史、画像缓存或个人资料。
- 正式包绝不包含 API Key、Token、Password、Cookie、账号或完整 WebDAV URL。
- 正式包构建必须从显式的干净 staging 目录生成。
- 内部测试包与正式包必须使用不同脚本、不同产物目录和不同文件名。
- 正式包脚本不得调用测试数据注入命令。
- 正式安装器不得默认删除用户数据目录。
- 升级、修复和卸载的数据保留策略必须明确并经过验证。

### 6.2 版本与产物

- 1.0 GA 的产品版本为 `1.0.0`。
- 必须建立一个可追踪的版本源，并同步到程序集、设置页、安装器、文件名和发布说明。
- 每个候选包必须带版本、架构和发布通道，避免测试包与正式包混淆。
- 正式产物必须生成 SHA-256。
- 发布报告必须记录安装包大小、安装后大小和主要体积来源。
- 是否签名必须明确。若 1.0 无代码签名证书，安装说明和发布说明必须准确说明，不得暗示已签名。

### 6.3 数据与数据库

- 应用首次启动继续由代码执行 EF Core migration。
- Phase 8 默认不新增 migration，也不执行 database update。
- 如果安全或发布修复必须改变数据库结构，先提交原因、影响面、迁移方案、兼容方案和回滚方案，等待明确授权。
- 正式安装器不得直接修改业务数据库。
- 升级前后的数据库兼容性必须用副本验证。
- 卸载默认不应删除用户数据库和缓存；如提供删除选项，必须显式、可理解且默认不选中。

### 6.4 文档真实性

- 每个操作步骤必须能追溯到当前代码入口或 RC 实机验证。
- 截图必须来自目标 RC，且不得包含私有媒体、账号、路径、URL 或密钥。
- 文档中的页面名、按钮名、状态名必须与 RC 一致。
- 外部服务要求必须区分必需、可选和仅对应功能需要。
- 网络不可用、API 未配置和媒体源不可访问时的降级行为必须准确描述。
- 不把开发环境启动方式写成正式用户安装方式。

### 6.5 日志与隐私

- 文档示例使用虚构且脱敏的数据。
- 不在 docs、README、发布日志或构建报告中写入完整本地路径、完整 WebDAV URL 或凭据。
- 正式发布前必须检查安装包、发布目录、文档、日志模板和脚本输出中的敏感信息。
- 若提供诊断收集方法，必须指导用户先脱敏。

## 9. 总体验收门槛

### 9.1 发布工程

- 正式包与测试包分离。
- 正式包不含用户数据和秘密。
- 正式安装器不破坏已有用户数据。
- 版本信息在应用、安装器和文档中一致。
- x64 正式包在支持的干净 Windows 环境安装、启动、运行、升级和卸载通过。
- 所有必需原生依赖随包完整且架构匹配。
- 产物可从记录的命令复现。
- SHA-256 和产物清单完整。

### 9.2 核心业务

- 本地目录和 WebDAV 配置、扫描、媒体库入库、详情、播放形成闭环。
- 电影与 TV 的识别、占位、修正和详情边界与 Phase 4～7 规则一致。
- 移出媒体库、删除记录、缓存清理均不删除真实本地或 WebDAV 文件。
- 播放器、音轨、外挂字幕、在线字幕、多播放源和续播完成 RC 验证。
- 收藏、想看、已看、不想看、历史和洞察符合既定数据语义。
- AI 和外部 API 不可用时，应用不会因文档承诺错误而被判定为核心不可用。

### 9.3 文档

- README、安装说明、使用说明书、帮助文档互相链接但不大段重复。
- 软件设计说明书与代码、现有设计资产和当前 UI 一致，并明确区分既有设计流程与 Phase 8 实施任务。
- 文档描述与 RC 一致。
- 所有敏感示例已脱敏。
- 所有外部服务都标注用途、是否可选和配置入口。
- 所有破坏性或容易误解的操作都有明确结果说明。
- Known Issues、支持范围和签名状态真实披露。

## 11. Known Issues 分类规则

### Blocker

满足任一条件即阻止 GA：

- 正式包包含秘密、私有数据或测试快照。
- 安装、升级或卸载可能无提示删除用户数据库或真实媒体文件。
- 正式包无法在支持平台启动或核心播放不可用。
- 版本或架构错误导致不可升级、不可卸载或加载错误原生依赖。
- 关键文档对删除、数据保留、外部服务或系统要求存在误导。
- 第三方组件许可不允许当前分发方式，或必要声明缺失。

### Deferred

不阻止 1.0，但必须披露并规划：

- 未纳入 1.0 的 ARM64。
- 暂无数字签名证书。
- 非核心的安装器视觉优化。
- 后续在线更新器、自动更新、崩溃上报或多语言文档。
- 不影响正确性的进一步体积优化。

### Noise

- 构建环境中的无害行尾警告。
- 首次准备打包工具的下载提示。
- 不进入正式产物的内部测试包和本地 artifacts。

## 12. 每个小阶段的固定输出

每个小阶段文档完成时必须记录：

- 阶段状态和完成日期。
- 目标与实际完成内容。
- 修改文件、新增文件、删除文件。
- 明确未做事项。
- 业务逻辑、数据模型、安装行为和文档行为是否变化。
- build、publish、installer 和实机验证结果；未执行的项目必须说明。
- migration diff 是否为空。
- 人工验收矩阵及结果。
- Known Issues：Blocker / Deferred / Noise。
- `git diff --stat`。
- 是否建议进入下一阶段。
- 建议 commit message。

## 13. Phase 8 结束标准

只有满足以下条件，Phase 8 才可结束：

- 8.1～8.9 文档全部更新为已完成或明确 Deferred。
- Blocker 为零。
- 正式 GA 包从干净 staging 构建。
- 正式包不含用户数据或秘密。
- 安装、首次启动、升级、修复、卸载均完成验证。
- 核心媒体工作流和播放器回归通过。
- 软件设计说明书、README、安装说明、使用说明书、帮助文档、发布说明和第三方声明完成并互相链接。
- 1.0.0 版本、架构、文件名和校验值一致。
- 最终报告包含完整 Git、migration、build、验证和 Known Issues 状态。
