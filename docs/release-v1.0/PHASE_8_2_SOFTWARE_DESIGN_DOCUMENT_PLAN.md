Last updated: 2026-06-20

## 文档定位

软件设计说明书面向开发、维护、测试和技术评审人员，描述 XFVerse 当前实际采用的整体设计、模块边界、数据结构、关键流程、安全约束、部署方式和 UI 设计。

它与其他文档的区别：

- 安装说明：面向终端用户，说明安装、升级、修复和卸载。
- 软件使用说明书：面向终端用户，说明功能和操作流程。
- 帮助文档：按问题和错误现象提供排查与恢复。
- 软件设计说明书：解释系统如何设计、模块如何协作，以及当前实现为何形成。

## 目标

1. 根据当前代码、项目文件、数据库模型和阶段文档，编写符合实际的软件整体设计说明书。
2. 记录 XFVerse 的总体架构、模块划分、数据设计、关键流程、安全隐私、性能、部署和发布设计。
3. 完整说明 UI 的信息架构、设计系统、页面结构、交互状态和实现方式。
4. 说明现有 UI 的实际设计流程：先绘制 UI 草图和页面规格，再依据草图修改实际 WPF UI，并在运行验收中持续收口。
5. 将现有 `DesignDraft` 草图、页面规格、Phase 7 计划与当前 XAML 建立可追溯关系。
6. 不把历史草图中未落地、已调整或与当前业务语义冲突的内容写成当前实现。

## 只做

- 审计当前解决方案、项目结构、核心服务、实体、ViewModel、View、原生依赖和打包代码。
- 阅读现有 `DesignDraft` 草图、页面规格和 UI 规则。
- 对照当前 XAML 和必要的实际运行截图，确认哪些设计已经落地、哪些经过调整、哪些未实现。
- 编写软件设计说明书。
- 绘制用于解释当前系统的架构图、数据图、流程图和页面信息架构图。
- 建立 UI 草图—页面规格—实际 View 的追溯矩阵。
- 记录设计决策、已知限制和 1.0 明确不做事项。

## 不做

- 不重新绘制一套新的 UI 草图。
- 不新增 UI 改造阶段。
- 不修改实际 XAML、ViewModel、Service 或业务代码。
- 不根据旧草图虚构当前不存在的功能。
- 不把设计说明书写成用户操作教程。
- 不新增 migration，不执行 database update。
- 不生成正式安装包。

### 12. UI / UX 设计

该章节说明当前 UI 的设计结果和形成过程。

#### 12.1 UI 设计流程

准确记录项目采用的流程：

1. 先在 `DesignDraft` 中绘制页面草图和目标效果图。
2. 为主要页面编写 `page-spec`、全局 UI 规则和资源说明。
3. 按阶段将草图转换为 WPF XAML、共享资源、控件和 ViewModel 状态。
4. 运行应用，对照草图检查布局、滚动、阴影、长文本、主题、弹窗和状态。
5. 当草图与业务语义、安全或实际数据能力冲突时，以业务和安全为先，并在阶段文档中记录取舍。
6. 通过多轮回归完成当前 UI 收口。

此处是对既有设计过程的说明，不是要求 Phase 8 重新走一遍设计和实现。

- 不能把阶段日志写成整体设计正文；正文应提炼稳定设计。
- 文档和图示不得包含私有路径、URL、媒体样本或凭据。
